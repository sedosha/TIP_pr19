# Практическое занятие №3. Логирование с помощью zap. Ведение структурированных логов

## Цель
Освоить организацию структурированного логирования в backend-приложении на Go с использованием библиотеки zap для записи, анализа и сопровождения событий приложения.

## Что было сделано
В проект микросервисов (Auth gRPC + Tasks HTTP) интегрирована библиотека **zap** для структурированного логирования. Логи выводятся в консоль в формате JSON, что позволяет анализировать их по ключевым полям.

**Zap** — это высокопроизводительная библиотека для структурированного логирования на Go, разработанная в Uber. Она поддерживает разные уровни логирования и позволяет добавлять к сообщениям структурированные поля в формате ключ-значение.

## Поля в логах проекта

### Основные поля (присутствуют во всех логах)

| Поле | Тип | Описание | Пример |
|------|-----|----------|--------|
| `level` | string | Уровень логирования | `"info"`, `"warn"`, `"error"`, `"debug"` |
| `ts` | float | Временная метка (Unix timestamp) | `1775746076.874709` |
| `msg` | string | Сообщение о событии | `"task created"`, `"incoming request"` |
| `caller` | string | Файл и строка, откуда вызван лог | `"http/handlers.go:66"` |

### Поля HTTP-запроса 

| Поле | Тип | Описание | Пример |
|------|-----|----------|--------|
| `request_id` | string | Уникальный идентификатор запроса | `"my-unique-request-123"` |
| `method` | string | HTTP метод | `"POST"`, `"GET"` |
| `path` | string | Путь запроса | `"/v1/tasks"`, `"/health"` |
| `remote_addr` | string | IP адрес клиента | `"[::1]:59160"` |
| `status_code` | int | Код ответа HTTP | `201`, `401`, `404`, `503` |
| `duration` | float | Время обработки в секундах | `0.002397214` |

### Поля в handlers

| Поле | Тип | Описание | Пример |
|------|-----|----------|--------|
| `task_id` | string | Уникальный идентификатор задачи | `"104ee8e9"` |
| `title` | string | Название задачи | `"Тестовая задача"` |
| `subject` | string | Имя пользователя из токена | `"user"` |

### Поля ошибок

| Поле | Тип | Описание | Пример |
|------|-----|----------|--------|
| `error` | string | Текст ошибки | `"connection refused"` |
| `stacktrace` | string | Стек вызовов (при error уровне) | Полный стектрейс |

### Создание задачи с request_id (POST /v1/tasks)

**Запрос:**
```bash
curl -X POST http://localhost:8090/v1/tasks \
  -H "Authorization: Bearer demo-token" \
  -H "Content-Type: application/json" \
  -H "X-Request-ID: my-unique-request-123" \
  -d '{"title":"Запрос с ID","description":"Проверка request_id","due_date":"2026-04-15"}'
```

**Ответ (201 Created):**
```json
{"id":"07a9b016","title":"Запрос с ID","description":"Проверка request_id","due_date":"2026-04-15","done":false,"created_at":"2026-04-09T17:54:46.939037281+03:00"}
```

**Логи (уровни info + debug):**
```json
{"level":"info","msg":"incoming request","request_id":"my-unique-request-123","method":"POST","path":"/v1/tasks"}
{"level":"debug","msg":"token validated","path":"/v1/tasks","subject":"user"}
{"level":"info","msg":"task created","request_id":"my-unique-request-123","task_id":"07a9b016","title":"Запрос с ID"}
{"level":"info","msg":"request completed","status_code":201,"duration":0.001487954}
```

**Вывод:** request_id `my-unique-request-123` прокидывается через все логи, связанные с этой операцией.

## Контрольные вопросы
**1. Зачем backend-приложению нужно логирование?**

Логирование необходимо backend-приложению для наблюдения за его состоянием в реальном времени и диагностики проблем. Без логов невозможно понять, что происходит внутри сервера — какие запросы приходят, какие ошибки возникают и сколько времени занимает обработка. Логи также помогают анализировать поведение пользователей и выявлять аномалии в работе системы.

**2. Чем обычный текстовый лог отличается от структурированного?**

Обычный текстовый лог представляет собой строку с произвольным форматированием, что затрудняет автоматический поиск и фильтрацию информации. Структурированный лог представляет собой набор пар ключ-значение, что позволяет легко искать события по любому полю, например, по коду ответа или идентификатору запроса. В отличие от текстового лога, структурированный можно анализировать программами без необходимости писать сложные регулярные выражения для каждого формата.

**3. Что означает structured logging?**

Structured logging — это подход к логированию, при котором каждое событие фиксируется в виде набора структурированных полей (ключ-значение) в дополнение к текстовому сообщению. Такой подход позволяет впоследствии эффективно фильтровать, агрегировать и анализировать логи с помощью инструментов вроде jq, Elasticsearch или Loki. Вместо того чтобы склеивать все данные в одну строку, каждое поле остается отдельным и доступным для поиска.

**4. Какие уровни логирования используются в этой работе?**

В данной работе используются четыре основных уровня логирования: Debug для детальной отладочной информации, Info для записи нормальных рабочих событий, Warn для предупреждений о нештатных ситуациях, когда сервис продолжает работу, и Error для фиксации ошибок, которые требуют внимания. Уровни помогают отделить важные события от второстепенных и позволяют в продакшене отключить debug-логи, чтобы не захламлять хранилище.

**5. Почему полезно логировать HTTP-метод, путь и статус ответа?**

Логирование HTTP-метода, пути и статуса ответа позволяет понять, какие именно эндпоинты вызываются и с каким результатом они завершаются. Если на сервер приходит много запросов с кодом 404, логи с путями покажут, какие маршруты не существуют, но клиенты пытаются к ним обратиться. Это помогает быстро обнаружить ошибки интеграции между сервисами или проблемы с клиентскими приложениями.

**6. Зачем в лог добавляют время выполнения запроса?**

Время выполнения запроса в логах помогает выявлять медленные эндпоинты и проблемы с производительностью. Если один и тот же запрос стал обрабатываться дольше обычного, это может указывать на проблемы с базой данных, сетью или алгоритмами внутри сервиса. Без метрики времени инженер может не заметить постепенной деградации производительности до тех пор, пока сервис не перестанет отвечать по таймаутам.

**7. Почему логирование ошибок должно содержать дополнительный контекст?**

Ошибки без контекста бесполезны для диагностики — простое сообщение «task not found» не говорит, какой именно идентификатор вызвал проблему и при каком запросе это произошло. Добавление дополнительных полей, таких как task_id, user_id или request_id, позволяет разработчику сразу понять причину ошибки и воспроизвести ситуацию. С контекстом ошибки становятся не просто записями в логе, а полноценными данными для расследования инцидента.

**8. В чём практическое преимущество zap?**

Практическое преимущество zap заключается в его высокой производительности и минимальных аллокациях памяти, что критически важно для высоконагруженных backend-приложений. Zap не использует отражения (reflection) для работы с полями, в отличие от многих других библиотек, что делает его значительно быстрее. При этом он предоставляет удобный API для структурированного логирования и гибкую настройку уровней и форматов вывода.

**9. Что означает maintenance mode у logrus?**

Maintenance mode у logrus означает, что библиотека больше не получает новых функций и находится только в режиме поддержки, то есть исправляются только критические ошибки и проблемы безопасности. Это произошло потому, что Go предоставил стандартный пакет slog, который решает те же задачи и считается предпочтительным для новых проектов. В сообществе Go рекомендуется переходить на slog или zap, а logrus считать устаревшим выбором.

**10. Почему structured logging особенно важен для микросервисов и backend API?**

Структурированное логирование особенно важно для микросервисов и backend API, потому что в распределённой системе один запрос может проходить через несколько сервисов, и без структурированных полей (особенно request_id) отследить его путь практически невозможно. В больших системах логи генерируются в огромных объёмах, и возможность фильтрации по полям становится не роскошью, а необходимостью для эксплуатации. Именно структурированные логи позволяют строить системы мониторинга и дашборды, которые автоматически агрегируют информацию об ошибках, времени ответа и частоте вызовов каждого эндпоинта.
